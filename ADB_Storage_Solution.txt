How It Actually Works
Unity Catalog's Role in File Storage
Unity Catalog Volumes are the right construct for unstructured files (images, PDFs, etc.). Managed volumes are fully managed by Unity Catalog, while external volumes represent data in ADLS registered in Unity Catalog to control and audit access from within Azure Databricks. Microsoft Learn
However, for external tables and volumes, Unity Catalog does not govern reads and writes performed directly against cloud object storage from external systems. You must configure additional policies and credentials in your Azure account to ensure that data governance policies are respected outside Azure Databricks. Microsoft Learn
This is the core limitation you need to plan around.

The Solution: Credential Vending (The Right Way)
The mechanism that lets external systems respect Unity Catalog ACLs without running Databricks compute is called Credential Vending.
Credential vending grants short-lived credentials using the Unity Catalog REST API. The granted credentials inherit the privileges of the Azure Databricks principal used to configure the integration. Path credential vending provides access to external locations in your Unity Catalog metastore. Microsoft Learn
Credential vending allows a catalog to grant temporary access to storage for an engine performing data processing. This is done through time-limited, downscoped storage credentials generated on demand, restricted to the specific storage needed for a higher-level object. The catalog manages both metadata and governance, while the engine only gets just-in-time access. Databricks
In practice for Azure, credential vending leverages Azure delegation SAS credentials Databricks — so your application calls the Unity Catalog REST API, gets a short-lived SAS token scoped to the permitted path, and uses that to access ADLS directly.
What's Required to Use Credential Vending
To receive a temporary credential for a path, the requesting Azure Databricks principal must be granted the EXTERNAL USE LOCATION privilege on the external location, and EXTERNAL USE SCHEMA privilege on the schema if accessing an external table. The Unity Catalog metastore must be explicitly enabled for external access. Microsoft Learn
External data access is disabled by default to prevent unauthorized external access. A metastore admin must enable it explicitly. Microsoft Learn

The key point: You must lock down the ADLS container so that only the Databricks managed identity has direct access. Users/apps cannot have direct ADLS RBAC roles — they must go through Unity Catalog's credential vending. Otherwise, the Unity Catalog ACLs are just advisory.

Important Limitations to Know

Not all external clients support credential vending, and support may vary depending on underlying cloud object storage. Databricks
Credential vending for Volumes (unstructured files) is newer than table credential vending — Databricks has planned to extend credential vending support to other Unity Catalog assets including volumes (unstructured data, arbitrary files). Databricks You should verify the current status for path-based vending in Azure specifically.
The cloud storage URLs generated by the credential vending API must be accessible to the external engine — ensure the engine is allowed by firewall and network access controls on the underlying ADLS accounts. Microsoft Learn


Bottom Line
✅ Yes, this architecture is viable — store files in ADLS, register them as Unity Catalog external volumes, govern access via UC ACLs, and let external apps obtain short-lived SAS tokens via the Credential Vending REST API instead of running Databricks compute.
⚠️ But you must remove direct ADLS access from all users/service principals except the Databricks managed identity, or the Unity Catalog governance is easily bypassed.



What is Keycloak?
Keycloak is an open-source Identity and Access Management (IAM) solution. It adds single sign-on (SSO) and authentication to applications and secures services with minimum effort, and is based on standard protocols — supporting OpenID Connect (OIDC), OAuth 2.0, and SAML. Keycloak
Think of it as a self-hosted alternative to Azure Entra ID (Azure AD) for handling authentication. Instead of every application managing its own login, users authenticate once through Keycloak, which then issues tokens that applications trust.
Applications are configured to point to and be secured by the Keycloak server. Browser applications redirect a user's browser to the Keycloak authentication server where they enter their credentials — users are completely isolated from applications and applications never see a user's credentials. Applications instead are given an identity token or assertion that is cryptographically signed, and these tokens can hold permission data so that applications can make authorization decisions. Keycloak
Key Keycloak capabilities

Single Sign-On (SSO) — log in once, access multiple systems
Identity Brokering — Keycloak can be configured to delegate authentication to one or more external identity providers, including social login via Facebook or Google, or any other OpenID Connect or SAML 2.0 IdP Keycloak
User Federation — Keycloak can validate credentials from external stores like LDAP or Active Directory and pull in identity information Keycloak
Fine-grained Authorization — role-based and attribute-based access control from a central admin console


Can Keycloak Be Used with Your ADLS + Unity Catalog Solution?
Yes — in a specific and valuable role. Keycloak fits into this architecture as the authentication layer that sits in front of the Unity Catalog credential vending flow. Here's how:
Role 1: SSO into Databricks (Officially Supported)
Databricks officially supports configuring Keycloak as the identity provider for single sign-on (SSO) in your Databricks account. Keycloak supports both OpenID Connect (OIDC) and SAML 2.0 for this integration. Databricks This means your users authenticate through Keycloak, and Databricks trusts those identities to enforce Unity Catalog permissions.
The limitation to know: Keycloak does not support SCIM to sync users and groups to Databricks Databricks, so you'd need to manage user provisioning separately.
Role 2: OIDC Federation for External Access (No Databricks Compute)
This is where Keycloak becomes powerful for your use case. Azure Databricks supports OIDC federation, allowing JSON Web Tokens (JWTs) issued by a recipient's identity provider to be used as short-lived OAuth tokens authenticated by Azure Databricks. This method is designed for recipients who do not have access to a Unity Catalog-enabled Databricks workspace. Microsoft Learn
Since Keycloak is a fully standards-compliant OIDC provider, it can serve as this IdP — your application authenticates with Keycloak, receives a JWT, and then exchanges it with the Databricks Unity Catalog API to get temporary SAS credentials for ADLS access.
The Databricks Delta Sharing service validates the JWT against the recipient's policy to ensure it matches the expected claims — issuer, audience, and subject. If validation is successful, the request is authenticated and access is granted based on Unity Catalog permissions. Microsoft Learn

When Would You Choose Keycloak?
Keycloak makes sense for this solution if:

You want self-hosted, open-source IAM with no dependency on Azure AD
You need to support non-Azure users (e.g., partners, customers, on-prem systems)
You're building a multi-tenant file access system where different organizations use different identity providers
Your organization already runs Keycloak for other systems and wants a single IAM layer

However, if your team is already on Microsoft Entra ID (Azure AD), it natively integrates with Databricks and you may not need Keycloak at all — Entra ID can serve the same OIDC federation role with less configuration overhead.
